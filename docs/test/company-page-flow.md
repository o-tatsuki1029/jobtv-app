# 企業ページフロー：要件・テスト項目・バグ候補

**この doc はテスト用。用語・テーブル・要件の公式な解釈は `docs/database-domain.md` の「企業ページ」セクションを参照すること。**

スタジオの企業ページ（`/studio/company`）、公開の企業ページ（`/company/[id]`）、および管理者による審査（`/admin/review` の企業ページタブ）について、テスト項目とバグ候補をまとめる。

---

## 1. 用語とデータの対応

| 用語 | テーブル | 説明 |
|------|----------|------|
| 企業基本情報 | `companies` | 企業名・ロゴ・業界・従業員数など。企業情報審査は `companies_draft`。本ドキュメントの主対象外。 |
| 企業ページ | `company_pages` / `company_pages_draft` | タグライン・会社紹介文・カバー画像・SNS・福利厚生など。スタジオで編集し、審査後に本番反映。 |

- **スタジオ「企業ページ管理」** = `company_pages_draft` の編集・保存・審査申請・公開トグル。
- **公開ページ** = `companies` + `company_pages`（`status = 'active'`）をマージして表示。
- **admin/review の「企業ページ」タブ** = `company_pages_draft` の `draft_status = 'submitted'` 一覧の承認・却下。

## 2. 要件一覧

### 2.1 スタジオ（`/studio/company`）

- リクルーターまたは管理者のみアクセス可能（レイアウトで `requireRecruiterOrAdmin()`）。
- 自社の企業ページドラフトを 1 件のみ編集する（`company_pages_draft`、自社 `company_id` に紐づく）。
- **表示**
  - キャッチコピー（必須・32 字以内）、会社紹介文（必須・3000 字以内）、カバー画像、SNS、福利厚生など。
  - ドラフトステータス（下書き / 審査中 / 承認済み / 却下）、本番の公開状態（公開中 / 非公開）を表示。
- **保存**
  - 保存時は `company_pages_draft` のみ更新。本番テーブルは変更しない。
  - 編集可能なのは `draft_status = 'draft'` のドラフト、または既存の submitted/approved/rejected を「編集モード」として同じドラフトを更新。
  - ドラフトが無い場合は新規作成。
- **審査申請**
  - 必須項目（キャッチコピー・会社紹介文）を満たしている場合のみ申請可能。
  - 申請で `draft_status` を `submitted` に更新。
  - 審査中は該当ドラフトの編集をロック（読み取り専用表示）。
- **公開トグル**
  - 承認済みドラフトが存在する場合のみ、本番の公開/非公開を切り替え可能（`company_pages.status` を `active` / `closed` に更新）。

### 2.2 公開ページ（`/company/[id]`）

- 認証不要で誰でも閲覧可能。
- **表示データ**
  - `companies` の 1 件 + `company_pages` の 1 件（`company_id` 一致かつ `status = 'active'`）をマージ。
  - 求人（`job_postings`, status=active）、動画（`videos`, status=active）、説明会（`sessions`, status=active）も取得して表示。
- **404 とする条件**
  - 企業（`companies`）が存在しない、または `getCompanyProfileById` がエラー/ null を返した場合。
  - `company_pages` が存在し、かつ `status !== 'active'` の場合（企業ページが「非公開」のとき）。
- **特例**
  - `id === 'uid'` のときはモックデータを表示（開発/デモ用）。

### 2.3 管理者審査（`/admin/review` の「企業ページ」タブ）

- 管理者のみアクセス可能（`requireAdmin()`）。
- **一覧**
  - `company_pages_draft` の `draft_status = 'submitted'` を取得し、企業名・説明・ロゴなどを表示。
- **プレビュー**
  - 審査対象ドラフトの内容をプレビュー用 URL（`/studio/company/preview-content`）で iframe 等で表示。親ウィンドウから postMessage でデータを渡す。
- **差分**
  - ドラフトと本番（`company_pages`）の差分を表示（変更がある項目を一覧）。
- **承認**
  - 承認時: ドラフトの内容で `company_pages` を更新または新規作成し、`status = 'active'` に設定。ドラフトの `draft_status` を `approved`、`production_page_id` を設定。
  - 同一 `company_id` で既存の `company_pages` があれば更新、無ければ新規作成。
- **却下**
  - ドラフトの `draft_status` を `rejected` に更新。本番テーブルは変更しない。

## 3. テスト項目リスト

### 3.1 スタジオ・企業ページ管理

- [ ] リクルーターでログインし `/studio/company` にアクセスできる。
- [ ] 未ログインで `/studio/company` にアクセスするとログインページにリダイレクトされる。
- [ ] 初回（ドラフトなし）でキャッチコピー・会社紹介文を入力して保存すると、新規ドラフトが作成される。
- [ ] 保存後、再度開いても同じ内容が表示される。
- [ ] キャッチコピー未入力で保存するとバリデーションエラーになる。
- [ ] 会社紹介文未入力で保存するとバリデーションエラーになる。
- [ ] キャッチコピー 32 字超過でエラーになる。
- [ ] 会社紹介文 3000 字超過でエラーになる。
- [ ] 必須を満たした状態で「審査に提出」できる。
- [ ] 審査申請後、ステータスが「審査中」になり、編集が読み取り専用になる。
- [ ] 審査中に保存ボタンを押しても保存されない（または無効）。
- [ ] プレビューを開くと、現在のドラフト内容がプレビュー画面に表示される。
- [ ] 承認済みで本番に `company_pages` がある場合、公開トグルで「公開中」「非公開」を切り替えられる。
- [ ] 非公開にすると、公開ページ `/company/[自社id]` が 404 になる。
- [ ] 公開中にすると、公開ページで内容が表示される。

### 3.2 公開ページ（`/company/[id]`）

- [ ] 存在する企業 ID かつ `company_pages` が active の場合、ページが表示される。
- [ ] 企業 ID が存在しない場合、404 になる。
- [ ] 企業は存在するが `company_pages` が無い場合、企業基本情報のみ表示される（または仕様に応じて 404）。
- [ ] 企業は存在するが `company_pages.status` が `closed` の場合、404 になる。
- [ ] 表示内容が、本番の `companies` と `company_pages`（active）のマージと一致する（タグライン・説明・カバー・SNS 等）。
- [ ] `/company/uid` のときはモックデータが表示される。

### 3.3 管理者審査（企業ページタブ）

- [ ] 管理者でログインし `/admin/review` を開き「企業ページ」タブを選択できる。
- [ ] 審査申請済み（`draft_status = 'submitted'`）の企業ページドラフトが一覧に表示される。
- [ ] 「プレビューを見る」でプレビュー用ページにドラフト内容が表示される。
- [ ] 「差分を確認」でドラフトと本番の差分が表示される（本番が無い場合は新規分のみ）。
- [ ] 承認を実行すると、該当ドラフトが「承認済み」になり、本番の `company_pages` が更新または作成され、公開ページに反映される。
- [ ] 却下を実行すると、ドラフトが「却下」になり、本番は変更されない。
- [ ] 承認後、スタジオ側で「審査中」が解け、公開トグルが利用可能になる（または承認済み表示になる）。

### 3.4 データ整合性・エッジケース

- [ ] 同一企業で複数ドラフトがある場合、スタジオでは「編集可能な 1 件」または「最新の承認済み/審査中/却下 1 件」のみ扱い、一貫して同じドラフトが表示・保存される。
- [ ] 承認時に `company_pages` が既に存在する場合は更新、無い場合は新規作成される。
- [ ] 承認後、`company_pages_draft.production_page_id` が正しく設定される。
- [ ] 公開トグルで「非公開」にしたあと、再度「公開」にすると、直近承認内容がそのまま表示される。

## 3.5 テスト項目の抜け漏れ確認（要件との対応）

以下は要件（Section 2）とテスト項目（3.1〜3.4）を照合した結果。未カバーまたは曖昧な点を「候補」として列挙する。必要に応じて 3.1〜3.4 に項目を追加すること。

| 要件・観点 | 現状のカバー | 抜け漏れ候補 |
|------------|--------------|--------------|
| スタジオ: 管理者でアクセス可能 | 3.1 でリクルーターのみ明示 | 管理者でログインし `/studio/company` にアクセスできることを確認する項目を追加してもよい。 |
| スタジオ: 一般ユーザー（候補者など）でアクセス不可 | 未ログインのリダイレクトのみ | リクルーター・管理者以外のロールでアクセスするとリダイレクトされることを確認する項目。 |
| スタジオ: 必須未満で審査申請不可 | 必須満たして提出はある | 必須未満のとき「審査に提出」が押せない／エラーになることを明示する項目。 |
| スタジオ: 却下後の再編集 | 3.4 で複数ドラフトに言及 | 却下済みドラフトを「編集モード」で開き、修正して再申請できることを確認する項目。 |
| スタジオ: 承認済みだが本番がまだ無い場合 | 3.1 で「本番に company_pages がある場合」と限定 | 承認直後（本番が作成された直後）に公開トグルが使えるようになる流れの確認。 |
| 公開: 求人・動画・説明会の表示 | 要件ではマージして表示とあるがテストに無し | 公開ページで、求人・動画・説明会のセクションが（データがある場合）表示されることを確認する項目。 |
| 公開: 企業あり・company_pages 無し | 3.2 で「企業基本情報のみ or 404」と二者択一 | 仕様確定用: どちらを正とするか doc で決め、その期待値のテスト 1 件。 |
| 審査: 未ログイン・非管理者でアクセス不可 | 3.3 で管理者で開けるのみ | 未ログインで `/admin/review` にアクセスするとリダイレクト。リクルーターでアクセスするとリダイレクト／403。 |
| 審査: 審査対象 0 件 | なし | 申請済みが 0 件のとき、企業ページタブで空状態が正しく表示されることを確認する項目。 |
| データ: 却下→再編集→再申請 | なし | 却下後に同じドラフトを編集して再度 submitted にできるか（運用上の期待に応じて追加）。 |
| データ: 承認／公開後の即時反映 | なし | 承認や公開トグル操作後、一覧・公開ページが再読み込みで最新になること（キャッシュ／revalidate の確認）。 |

## 4. バグ候補（コード確認に基づく）

以下は実装を読んだ範囲での「要確認」ポイント。テストまたはランタイムで確認推奨。

### 4.1 公開ページの 404 条件

- **箇所**: `app/(main)/company/[id]/page.tsx`
- **内容**: `company_pages` にレコードが**無い**場合、`companyPageCheck` は `null`。このとき `companyPageCheck && companyPageCheck.status !== 'active'` は false のため 404 にせず、`companies` の情報だけをマージして表示する。
- **懸念**: 企業ページを一度も作成・承認していない企業は「企業ページ」が無いが、公開 URL では企業名など基本情報だけ表示される。仕様として「企業ページが無い場合は 404」にしたいなら条件の見直しが必要。

### 4.2 getCompanyProfileById のマージ順

- **箇所**: `lib/actions/company-profile-actions.ts` の `getCompanyProfileById`
- **内容**: `result = { ...companyData, ...(pageResult.data || {}), job_postings, videos, sessions }` で、`company_pages` のフィールドが `companies` の同名フィールドを上書きする。
- **懸念**: `companies` と `company_pages` で同名カラム（例: 将来的に `description` が両方に存在するなど）があると意図しない上書きがあり得る。現状のスキーマでは問題ないが、スキーマ変更時に注意。

### 4.3 審査ページのプレビューと draftId

- **箇所**: `admin/review` の企業ページプレビューでは `onPreview(draftId)` で `draftId` を渡し、プレビュー用コンテンツを取得している。
- **確認**: 企業ページタブのプレビューが、`getCompanyPageDraftByIdAdmin(draftId)` 等で正しいドラフトを取得し、`CompanyProfileView` 用のデータに変換できているか。他タブ（求人・説明会）と同様のパターンになっているか。

### 4.4 承認時の production_page_id と UNIQUE

- **箇所**: `admin-actions.ts` の `approveCompanyPage`
- **内容**: 承認後に `company_pages_draft` の `production_page_id` を更新する。既に同じ `production_page_id` を持つ別ドラフトがある場合はそのドラフトを削除している。
- **懸念**: 削除したドラフトをスタジオ側が参照している場合、表示不整合が起きる可能性。通常は「承認済み 1 件」のみが production_page_id を持つ想定なので、運用で問題ないか確認推奨。

### 4.5 モック id "uid" の残存

- **箇所**: `app/(main)/company/[id]/page.tsx`
- **内容**: `id === 'uid'` のときモックデータを返す。
- **懸念**: 本番で `/company/uid` にアクセスするとモックが表示される。本番では無効化するか、環境変数で制御する方が安全。

### 4.6 スタジオの「企業ページ」と「企業情報」の混同

- **内容**: スタジオには「企業ページ管理」（`/studio/company` = company_pages_draft）のほか、企業の基本情報を編集する画面（例: 設定のプロフィールや企業情報）がある可能性がある。審査側では「企業情報」（companies_draft）と「企業ページ」（company_pages_draft）が別タブ。
- **確認**: 要件として「企業ページ」のみを対象にしているため、企業情報（companies_draft）の作成・申請フローが別画面で存在する場合、その画面と審査タブの対応が正しいか。
